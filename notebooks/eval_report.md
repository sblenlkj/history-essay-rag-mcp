# Eval evaluation · Agent demo

Файл анализирует прогон eval-набора из 20 запросов для проекта RAG + MCP.

Проверка делалась по требованиям домашнего задания: нужно показать top-k поиск через MCP/агента, возвращение фрагментов с `document_id`, `chunk_id`, `source` и провести ручную оценку на 15–20 запросах. В этом eval-наборе проверяются смысловой поиск, поиск по metadata автора цитаты, поиск с `interpretation_type`, широкие запросы, опечатки и negative/no-answer сценарии.

## Общие впечатления

В целом система работает успешно. Агент вызывает MCP-инструменты, получает результаты из Qdrant-backed search service и в большинстве случаев возвращает релевантные темы с устойчивыми идентификаторами. Особенно хорошо отработали:

- поиск по автору цитаты (`author_metadata`);
- запросы с явным типом интерпретации, например `negative_assessment`, `comparative`, `mixed_assessment`;
- широкие смысловые запросы по историческим периодам и процессам;
- негативные запросы, где в корпусе нет подходящей темы.

Главное сильное место — агент почти всегда показывает нужные идентификаторы: `document_id`, `chunk_id`, `source`, а также автора и текст темы. Это важно для доверия к RAG-ответу: результат можно проверить и воспроизвести.

Главные ограничения:

1. В нескольких случаях агент выдаёт частично релевантные результаты, когда запрос слишком широкий или требует тонкой тематической фокусировки.
2. В части ответов агент начинает формулировать собственный заголовок темы или краткое описание перед исходным `content`. Это не критично, если ниже сохраняется исходный фрагмент, но для строгого RAG лучше опираться на поле `content`.
3. Запрос про причины и предпосылки реформ Петра I оказался самым слабым: выдача частично ушла в Екатерину II и послепетровский период.
4. Автоматически определить реальный tool call из сохранённого JSON нельзя, потому что в eval-results сохранён только финальный ответ агента. Поэтому ручная оценка ниже сфокусирована на релевантности результатов, наличии идентификаторов и соответствии ожидаемому поведению.

## Итоговый вывод

Eval-набор из 20 запросов покрывает основные случаи, которые требуются в задании: широкие смысловые вопросы, metadata-запросы, запросы по автору, запросы по типу интерпретации, неоднозначные запросы, опечатки и вопросы без хорошего ответа.

Система в целом выполняет ключевое требование: агент через MCP-инструменты возвращает top-k фрагменты с `document_id`, `chunk_id`, `source` и текстом. Большинство результатов релевантны, а negative/no-answer сценарии показывают, что агент не всегда пытается выдать нерелевантные документы как точный ответ.

Наиболее слабые места:

1. Причинно-следственные запросы требуют более точной настройки retrieval или отдельной логики для `causal_explanation`.
2. Для некоторых широких запросов выдача полезная, но частично смешивает соседние темы.
3. Формат ответа агента не полностью детерминирован: иногда он добавляет собственный заголовок или краткую формулировку. Поэтому для строгой проверки лучше анализировать сохранённый JSON и результаты MCP tool, а не только финальный prose-ответ.

Общая ручная оценка: **16 good / 4 partial / 0 bad**. Для учебного RAG + MCP проекта результат можно считать успешным.
